## 1. Purpose

This synthesis records the first independent review cycle for **P-0001: AI-Lab Documentation Architecture Proposal v0.1**.

It does not replace the original reviews.

Instead, it identifies:

- convergences between reviewers,
- major disagreements or tensions,
- accepted corrections,
- deferred questions,
- and concrete requirements for **P-0001 v0.2**.

This document exists to prevent the review process itself from becoming lossy. The critiques that shaped the next proposal version should remain visible, traceable, and inspectable.

## 3. Summary of Review Outcome

Both reviewers considered the direction of P-0001 valuable, but both identified the same deep structural risk:

> P-0001 v0.1 risks treating documents as the durable unit of knowledge.

This conflicts with the emerging AI-Lab principle that documents are carriers, views, or artifacts, while durable knowledge should eventually live in a deeper structure of claims, questions, evidence, warrants, relationships, provenance, disputes, and revisions.

The synthesis conclusion is:

> P-0001 should be revised, not rejected.

P-0001 v0.2 must explicitly define itself as a proposal for the **document layer**, not the entire knowledge layer.

---

## 4. Major Convergence: Documents Are Not Knowledge Atoms

### 4.1 Claude Review

Claude argued that P-0001 organizes documents while the durable unit was supposed to be the claim. According to this critique, ALS, ADR, P, R, and SYN are document types, but the deeper AI-Lab commitment is to preserve traceable units such as claims, questions, evidence, warrants, and relationships.

Claude’s central warning:

> Classification, identity, and relationships have been encoded into locators and prose instead of being kept as revisable data.

### 4.2 Gemini Review

Gemini identified the same risk under a different name: **Document Isolation**.

Gemini warned that if the project is mapped only as a hierarchy of text files, the documentation may eventually become a graveyard of contradictory statements that no future human or AI peer can reliably trace.

Gemini’s central correction:

> The documentation architecture must eventually be treated as an instance of the AI-Lab knowledge graph itself.

### 4.3 Synthesis

These critiques independently converge.

Accepted correction:

> Documents are not the atomic unit of AI-Lab knowledge. Documents are human-readable carriers and bootstrap graph nodes. Their relationships must be made explicit, queryable, and eventually machine-readable.

**Decision:** Accepted with high confidence.

---

## 5. Major Convergence: A Document Layer Must Be Explicit

P-0001 v0.1 failed to clearly state that it describes a document layer that sits above a deeper knowledge layer.

P-0001 v0.2 must introduce at least three distinct layers:

```text
Knowledge Layer
  Claims
  Questions
  Evidence
  Warrants
  Relationships
  Provenance
  Disputes
  Revisions

Document Layer
  ALS
  ADR
  P
  R
  SYN
  Human-readable artifacts

Storage Layer
  Files
  Directories
  Git
  Databases
  Object stores
```

The document layer should be treated as a transitional and human-readable representation of the knowledge layer, not as a replacement for it.

**Decision:** Accepted with high confidence.

---

## 6. Major Convergence: Relationships Must Not Hide in Folders or Prose

Both reviewers objected to relationship semantics being encoded only through filenames, directory placement, or narrative text.

Examples of fragile hidden relationships:

```text
R-0001 reviews P-0001
SYN-0001 synthesizes R-0001 and R-0002
P-0001 v0.2 supersedes P-0001 v0.1
ADR-0005 implements ALS-0003
A review disputes a specific claim in a proposal
```

In P-0001 v0.1, these relationships are implied by filenames or prose.

P-0001 v0.2 must require explicit relationship metadata.

Example:

```yaml
relations:
  reviews:
    - P-0001:v0.1
  synthesizes:
    - R-0001
    - R-0002
  supersedes: []
  disputed_by: []
  depends_on: []
```

**Decision:** Accepted with high confidence.

---

## 7. Major Convergence: Avoid Semantic IDs and Semantic Paths

Claude strongly objected to identifiers such as:

```text
ALS-000 = philosophy
ALS-100 = epistemology
ALS-200 = ontology
```

because these encode mutable classification into permanent identity.

Gemini also warned against relying on ordinary directory structure as the true system of provenance.

### 7.1 Accepted Principle

Permanent identity should not carry classification that may later change.

### 7.2 Implication

Prefer opaque or minimally semantic stable IDs:

```text
ALS-0001
ALS-0002
ALS-0003
P-0001
R-0001
SYN-0001
ADR-0001
```

Category should be metadata:

```yaml
category: philosophy
layer: document-layer
document_type: specification
```

rather than being embedded into the identifier or path.

### 7.3 Directory Guidance

Directories may be used for human convenience, but must not be the authoritative source of classification.

**Decision:** Accepted with high confidence.

---

## 8. Major Convergence: Git History Is Not Sufficient Provenance

Gemini warned against relying on Git history for long-term scientific provenance. Git is an infrastructure tool, not the enduring provenance layer of AI-Lab.

### 8.1 Accepted Principle

Frozen and superseded documents should remain explicitly accessible as artifacts.

### 8.2 Implication

A frozen version should be preserved through explicit version identity and content metadata, not merely through Git history.

Possible future pattern:

```text
document_id: P-0001
version: 0.1
status: superseded
content_hash: <future hash>
superseded_by: P-0001:v0.2
```

The exact storage pattern remains deferred.

**Decision:** Accepted with high confidence.

---

## 9. Major Convergence: Documentation Must Bootstrap the Future Graph

Gemini posed the bootstrapping question:

> How should AI-Lab handle the initial phase where files are needed before the graph engine exists?

The synthesis answer is:

> Use graph-shaped Markdown.

In the bootstrap phase, Markdown files should include structured front matter describing identity, metadata, relations, contributors, status, and open tensions.

This allows the files to be human-readable now and graph-importable later.

Example future front matter:

```yaml
---
id: P-0001
title: AI-Lab Documentation Architecture Proposal
version: 0.2-draft
status: proposal
document_type: proposal
layer: document-layer
category: documentation-architecture

relations:
  supersedes:
    - P-0001:v0.1
  reviewed_by:
    - R-0001
    - R-0002
  synthesized_by:
    - SYN-0001

contributors:
  - peer_id: PEER-MUSTAFA-AKSU
    role: project_lead
    substrate: human
  - peer_id: PEER-GPT-ARCHITECT
    role: drafter
    substrate: GPT-5.5 Thinking

open_tensions:
  - document_layer_vs_knowledge_layer
  - graph_native_design_vs_bootstrap_files
---
```

**Decision:** Accepted with high confidence as a bootstrap approach.

## 10. Major Correction: Separate Research Artifacts from Framework Documentation

Gemini argued that putting RTG papers, DSNs, experiments, and notebooks directly into the AI-Lab framework documentation tree risks binding AI-Lab to one research domain too early.

Claude also warned that DSNs and papers are not merely project documentation; they are scientific artifacts.

### 10.1 Accepted Correction

The AI-Lab framework repository should distinguish:

```text
AI-Lab framework specifications
  How AI-Lab defines itself

Research program artifacts
  Scientific outputs produced using AI-Lab
```

### 10.2 Deferred Question

It remains unresolved whether RTG artifacts should live:

- in a separate repository,
- in a separate AI-Lab-managed research graph,
- or in a clearly separated `research-programs/` area outside framework documentation.

**Decision:** Accepted with high confidence; exact repository boundary deferred.

---

## 11. Major Correction: Preserve Disagreement Structurally

Claude warned that the lifecycle:

```text
Review → Synthesis → Frozen
```

can accidentally erase dissent if the frozen document only reflects what the synthesis absorbed.

Gemini similarly warned against consensus bias and recommended support for epistemic forking.

### 11.1 Accepted Principle

Disagreement must travel with the conclusion it contested.

A frozen document should structurally point to:

- reviews that challenged it,
- dissenting claims,
- unresolved tensions,
- alternative branches,
- and deferred objections.

### 11.2 Epistemic Forking

If synthesis cannot honestly reconcile two warranted perspectives, the documentation architecture should allow multiple coexisting branches.

Example:

```text
ALS-0005-A
ALS-0005-B
```

or a future branch relationship:

```yaml
relations:
  forks_from:
    - ALS-0005:v0.1
  competes_with:
    - ALS-0005-B:v0.1
```

The exact naming convention is deferred.

**Decision:** Accepted in principle. Forking mechanism deferred.

## 12. Correction: Orthogonal Status, Confidence, and Warrant Axes

Claude identified vocabulary collision between:

- document maturity,
- confidence level,
- claim warrant,
- DSN-style tags.

P-0001 v0.2 must keep these axes separate.

### 12.1 Document Maturity

Describes lifecycle state of a document.

Example candidates:

```text
proposal
draft
reviewed
synthesized
frozen
superseded
retired
```

### 12.2 Confidence

Describes reviewer confidence in a recommendation.

Example candidates:

```text
high_confidence
moderate_confidence
speculative
```

### 12.3 Claim Warrant

Describes epistemic status of a claim.

This should not be finalized in P-0001. It belongs more properly to a later epistemology or ontology document.

**Decision:** Accepted with moderate-high confidence.

---

## 13. Correction: Contributor Attribution Must Separate Peer Identity from Substrate

Claude warned against attributing work only to model names such as “Claude” or “GPT-5.5 Thinking,” because this binds provenance to substrate rather than persistent research identity.

Gemini argued for a unified contributor structure rather than separate human and machine provenance sections.

### 13.1 Accepted Principle

Contributors should be represented through unified contributor records.

Each record should distinguish:

- persistent peer identity,
- role,
- substrate or instantiation,
- date/context of contribution.

Example:

```yaml
contributors:
  - peer_id: PEER-MUSTAFA-AKSU
    display_name: Mustafa Aksu
    kind: human
    role: project_lead

  - peer_id: PEER-GPT-ARCHITECT
    display_name: GPT Architect
    kind: ai_peer
    substrate: GPT-5.5 Thinking
    role: drafter

  - peer_id: PEER-CLAUDE-REVIEWER
    display_name: Claude Ontology Reviewer
    kind: ai_peer
    substrate: Claude
    role: independent_reviewer
```

**Decision:** Accepted with moderate-high confidence.

---

## 14. Correction: Keep the Lightweight Path Cheap

Claude warned that P-0001 risks creating too much ceremony too early.

This is accepted.

### 14.1 Accepted Principle

The system should distinguish between lightweight and heavyweight artifacts.

Lightweight:

```text
P — Proposal
R — Review
Notes
```

Heavyweight:

```text
ALS — Specification
SYN — Synthesis
Frozen documents
```

The lightweight path must remain cheap enough that real reasoning does not stay trapped in chat because formalization feels too expensive.

**Decision:** Accepted with moderate confidence.

---

## 15. Correction: Treat Documentation Architecture as Adopted Process, Not Discovered Structure

Claude warned that P-0001 borrows from existing governance patterns such as RFCs and ADRs, and should not mistake this process convention for a discovered structure of inquiry.

Accepted wording for P-0001 v0.2:

> The proposed documentation architecture is a designed process, inspired by scientific review, standards practice, and software architecture records. It is not claimed to be the final or natural structure of inquiry.

**Decision:** Accepted with moderate confidence.

## 16. Deferred Questions

The following are not resolved by this synthesis and should be carried into P-0001 v0.2 as open questions or open tensions.

### 16.1 Exact ID Format

Should IDs be fully opaque, minimally semantic, or family-prefixed?

Candidate:

```text
P-0001
R-0001
SYN-0001
ALS-0001
ADR-0001
```

### 16.2 Exact Directory Structure

Should directories be flat, family-based, or category-based?

Accepted constraint:

> Paths are locators, not authoritative taxonomy.

### 16.3 Exact Catalog Format

Should the document relationship catalog be:

- YAML,
- JSON,
- Markdown table,
- generated from front matter,
- or later stored in a database?

### 16.4 Exact Frozen Version Storage

Should frozen versions live:

- beside active documents,
- in a history folder,
- under versioned folders,
- or as content-addressed artifacts?

### 16.5 SYN as Separate Family

Should SYN remain a distinct document family, or be treated as a special type of review?

Current leaning:

> Keep SYN separate for now, because it answers a different question than a review.

### 16.6 Research Program Boundary

Should RTG research artifacts be stored in:

- the AI-Lab repository,
- a separate RTG repository,
- a future AI-Lab research graph,
- or a clearly separated research-programs area?

### 16.7 Machine-Readable Ontology Documents

Should ontology and epistemology documents eventually become dual-aspect files: both human-readable prose and machine-readable schema?

This is accepted as important but remains speculative.

## 17. Requirements for P-0001 v0.2

P-0001 v0.2 must include the following changes.

### 17.1 Add Layer Boundary

Explicitly distinguish:

```text
Knowledge Layer
Document Layer
Storage Layer
```

### 17.2 Define Documents as Carriers

State clearly that documents are not the atomic knowledge unit.

### 17.3 Introduce Graph-Shaped Markdown

Require structured metadata/front matter for bootstrap documents.

### 17.4 Replace Semantic Numbering

Avoid ID schemes that encode category into permanent ID.

### 17.5 Move Classification to Metadata

Use category/layer/type metadata rather than semantic folder paths as authoritative classification.

### 17.6 Add Relationship Metadata

Require documents to declare structural relationships such as:

```text
reviews
synthesizes
supersedes
depends_on
implements
critiques
disputes
forks_from
```

### 17.7 Separate Status Axes

Separate document maturity, confidence, and claim warrant.

### 17.8 Preserve Frozen Versions Explicitly

Do not rely only on Git history.

### 17.9 Support Epistemic Forking

Allow unresolved competing branches where synthesis cannot honestly converge.

### 17.10 Separate Framework Docs from Research Outputs

Do not collapse AI-Lab framework documentation and RTG scientific artifacts.

### 17.11 Add Contributor Identity/Substrate Distinction

Use contributor records that separate persistent peer identity from substrate/model instance.

### 17.12 Keep Lightweight Paths Cheap

Avoid making the process so heavy that contributors avoid documenting reasoning.

## 18. Accepted Design Direction

The revised documentation architecture should be understood as:

> A bootstrap document graph for AI-Lab, implemented initially as structured Markdown, designed to preserve provenance and relationships until a deeper knowledge graph exists.

This is the central design shift from P-0001 v0.1 to P-0001 v0.2.
