## 1. Purpose

This synthesis records the second independent review cycle for **P-0001: AI-Lab Documentation Architecture Proposal v0.2**.

It identifies the remaining hard-to-repair architectural risks before the documentation architecture is implemented.

This document does not replace the raw reviews. It summarizes their convergence, preserves their disagreements, and defines the requirements for **P-0001 v0.3**.

## 3. Overall Review Result

Both reviewers judged P-0001 v0.2 as a major improvement over v0.1.

The first-round problems were mostly corrected:

- document layer vs knowledge layer was clarified;
- semantic category numbering was reduced;
- paths were demoted to locators;
- Git history was rejected as sole provenance;
- synthesis consensus bias was mitigated by epistemic forking;
- contributor records separated kind, role, and substrate;
- graph-shaped Markdown was accepted as a reasonable bootstrap strategy.

However, both reviewers identified unresolved issues in the same design region:

> Identity and relationships.

P-0001 v0.3 should therefore focus narrowly on:

1. immutable identifiers,
2. append-only relation records,
3. canonical relation schemas,
4. generated catalogs,
5. and the boundary between frozen nodes and evolving graph topology.

---

## 4. Major Convergence: Frozen Nodes Require Mutable Edge Topology

Gemini identified the **Frozen State Mutation Paradox**.

If a document is frozen, its content cannot be modified. But future documents may later dispute, refine, or supersede it. If the disputed frozen document cannot receive new backlinks, the graph becomes unidirectional and partially blind.

Claude independently identified the related integrity issue: relationships are stored twice, in front matter and in a catalog, without a single authoritative source or inverse-edge discipline.

### Synthesis

The shared correction is:

> Frozen documents should be immutable nodes, but the graph around them must remain extendable through append-only relation records.

This means P-0001 v0.3 must not rely solely on document front matter for all relationships.

**Decision:** Accepted with high confidence.

---

## 5. Accepted Design: Relation Records / Edge Artifacts

P-0001 v0.3 should introduce a first-class document-layer artifact for relationships.

Working name:

```text
Relation Record
```

Possible ID pattern:

```text
EDGE-0001
```

A relation record is an append-only artifact that declares a relationship between two or more nodes.

Example:

```yaml
---
id: EDGE-0001
document_type: relation_record
status: active

subject:
  id: P-0042
  version: 0.1-draft

predicate: disputes

object:
  id: ALS-0002
  version: 1.0-frozen

warrant:
  summary: "P-0042 identifies a conflict with assumptions used in ALS-0002."
  evidence:
    - P-0042

contributors:
  - peer_id: PEER-0001
    role: relation_author
    substrate: human
---
```

This allows the graph to evolve without mutating frozen documents.

**Decision:** Accepted with high confidence.

---

## 6. Major Convergence: Catalog Must Be Derived, Not Hand-Maintained Truth

Claude warned that v0.2 duplicates relationships in both front matter and `documents.yaml`.

Gemini warned that relying on a monolithic catalog would make the graph brittle.

### Synthesis

The catalog should be treated as a generated or derived index.

It should not be the authoritative source of relationships.

Preferred bootstrap discipline:

```text
Authoritative sources:
  - document front matter for document identity and local outgoing relations while mutable
  - append-only relation records for relationships involving frozen or external nodes

Derived sources:
  - documents.yaml
  - navigation indexes
  - rendered graph views
```

**Decision:** Accepted with high confidence.

---

## 7. Major Correction: Canonical Relation Schema Required

Claude identified an immediate consistency bug in P-0001 v0.2:

- the document header uses one relation shape;
- Section 8.1 uses another relation shape;
- some relation-like entries point to freeform labels rather than resolvable nodes.

P-0001 v0.3 must define one canonical relation shape.

Preferred direction:

```yaml
relations:
  - predicate: reviews
    target:
      id: P-0001
      version: 0.2-draft
    warrant:
      summary: "Independent review requested for second-round critique."
```

Flat relation arrays should be avoided for important relationships.

**Decision:** Accepted with high confidence.

---

## 8. Major Correction: IDs as Immutable Birth-Stamps

Claude identified a remaining identity flaw:

- `P-`, `ALS-`, `ADR-` prefixes encode document type into permanent IDs;
- `PEER-GPT-ARCHITECT` encodes substrate/vendor into peer identity.

This is dangerous because document type and substrate can change.

### 8.1 Accepted Principle

Identifiers are immutable birth-stamps, not live classifications.

An ID assigned at birth never changes.

If a proposal is later promoted to a specification, its ID remains the same. The current document type is expressed only in metadata.

Example:

```yaml
id: P-0001
origin_type: proposal
current_type: specification
```

For peer identities:

```yaml
peer_id: PEER-0001
display_name: GPT Architect
kind: ai_peer
substrate: GPT-5.5 Thinking
```

The peer ID should not encode the substrate.

**Decision:** Accepted with high confidence.

---

## 9. Reconciliation with Human Legibility

Fully opaque IDs may be more robust, but they reduce human usability during the bootstrap phase.

The synthesis accepts a middle position:

```text
Family-prefixed IDs may be retained as origin birth-stamps,
but must never be interpreted as current type.
```

Therefore:

```text
P-0001
```

means:

> The artifact originated as a proposal.

It does not permanently mean:

> The artifact is currently only a proposal.

This resolves the promotion problem without immediately requiring fully opaque IDs.

**Decision:** Accepted with moderate-high confidence.

---

## 10. Contributor Identity Fix

Current peer IDs such as:

```text
PEER-GPT-ARCHITECT
PEER-CLAUDE-REVIEWER
PEER-GEMINI-REVIEWER
```

still encode substrate or vendor.

P-0001 v0.3 should change examples to substrate-independent peer IDs:

```text
PEER-0001
PEER-0002
PEER-0003
PEER-0004
```

with display names and substrate in metadata:

```yaml
peer_id: PEER-0002
display_name: Architect Peer
kind: ai_peer
substrate: GPT-5.5 Thinking
role: drafter
```

**Decision:** Accepted with high confidence.

---

## 11. Dual-Aspect Ontology Documents

Gemini recommended that ontology and epistemology documents eventually become dual-aspect: both human-readable and machine-parseable.

This is important, but it does not need to be solved in P-0001.

P-0001 v0.3 should preserve it as an open tension or future requirement.

**Decision:** Deferred as speculative but important.

---

## 12. Requirements for P-0001 v0.3

P-0001 v0.3 must make the following changes:

1. Add **Relation Records / Edge Artifacts** as append-only relationship carriers.
2. Define the frozen-node / mutable-edge distinction.
3. State that catalogs are derived indexes, not authoritative graph truth.
4. Define one canonical relation schema.
5. Avoid freeform pseudo-edge labels that do not resolve to nodes.
6. Define IDs as immutable birth-stamps.
7. Clarify that family prefixes are origin markers, not current classifications.
8. Replace substrate-encoded peer IDs with substrate-independent peer IDs.
9. Define inverse-edge discipline: inverse edges should be derivable, not hand-authored twice.
10. Keep dual-aspect ontology/epistemology documents as a deferred requirement.

## 14. Closing Note

P-0001 v0.2 succeeded in correcting the first-order layer confusion.

The second review cycle revealed the next-order structure:

> AI-Lab documents are immutable or mutable nodes; relationships are append-only, warranted, independently addressable edge artifacts.

This is a significant architectural discovery.

It moves the documentation system closer to the deeper AI-Lab thesis:

> Understanding is preserved not by files alone, but by warranted, traceable, revisable relationships among persistent artifacts.
