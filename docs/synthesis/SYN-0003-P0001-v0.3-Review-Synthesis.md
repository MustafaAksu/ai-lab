## 1. Purpose

This synthesis records the third review cycle for **P-0001: AI-Lab Documentation Architecture Proposal v0.3**.

It synthesizes the reviews from Claude and Gemini, identifies the remaining edge-layer issue, and defines the decision required before repository implementation.

## 3. Overall Review Result

Both reviewers judged P-0001 v0.3 as a successful resolution of the node-level identity and relation problems found in v0.2.

Accepted as resolved:

- document-layer / knowledge-layer distinction,
- birth-stamp document IDs,
- substrate-independent peer IDs,
- canonical object-shaped relation metadata,
- need for append-only relation records,
- generated catalog discipline,
- frozen-node / mutable-graph distinction.

Both reviewers also judged append-only relation records as necessary rather than excessive.

The remaining issue is not a regression. It is the next layer down:

> Once edges become first-class artifacts, edges need the same identity, immutability, and addressing discipline as documents.

---

## 4. Major Convergence: Edge Records Are Artifacts Too

Claude observed that P-0001 v0.3 gives documents disciplined immutable identity, but does not fully apply the same discipline to edge records.

Gemini independently observed that the proposal is incomplete around edge lifecycle semantics and version propagation.

The synthesis conclusion:

> Edge records are artifacts. Therefore, edge records must have stable identity, immutable content, explicit scope, and well-defined correction semantics.

**Decision:** Accepted with high confidence.

---

## 5. Accepted Principle: Edges Are Immutable and Append-Only

A relation record should not be mutated after creation.

If an edge is wrong, outdated, refined, disputed, resolved, or nullified, AI-Lab should create a new edge that targets the earlier edge.

Do not mutate:

```yaml
id: EDGE-0001
status: retracted
```

Prefer:

```yaml
id: EDGE-0020
predicate: retracts
object:
  id: EDGE-0001
```

This keeps the historical graph append-only.

**Decision:** Accepted with high confidence.

---

## 6. Accepted Principle: Edges Can Target Edges

If relationships are first-class, then relationships must themselves be disputable.

Therefore `subject` and `object` in an edge record must be able to reference any artifact type, including:

```text
P
ALS
ADR
R
SYN
EDGE
future claim IDs
future artifact IDs
```

This completes the reification logic.

**Decision:** Accepted with high confidence.

---

## 7. Accepted Principle: References Need Scope

Gemini identified the edge inheritance problem.

Sometimes a relation targets:

```text
a specific version of an artifact
```

Sometimes it targets:

```text
the artifact lineage as a whole
```

P-0001 v0.3 only clearly showed version-pinned references.

P-0001 v0.4 or ADR-0003 must add explicit reference scope.

Candidate schema:

```yaml
object:
  id: ALS-0002
  scope: version
  version: 1.0-frozen
```

or:

```yaml
object:
  id: ALS-0002
  scope: lineage
```

Meaning:

- `scope: version` applies only to the specified version.
- `scope: lineage` applies to the artifact identity across versions until resolved, narrowed, or nullified by a later edge.

**Decision:** Accepted with high confidence.

---

## 8. Accepted Principle: Edge Correction Is Edge Creation

An edge should not be edited to change its meaning.

Instead, use new relation records such as:

```text
disputes
retracts
resolves
nullifies
supersedes
narrows
broadens
qualifies
```

Example:

```yaml
id: EDGE-0021
predicate: resolves
subject:
  id: ALS-0002
  scope: version
  version: 2.0-draft
object:
  id: EDGE-0007
  scope: exact
warrant:
  summary: "ALS-0002 v2.0 removes the flawed assumption identified by EDGE-0007."
```

**Decision:** Accepted with high confidence.

---

## 9. Edge Storage: Individual Files vs Ledgers

Gemini recommended considering append-only edge ledger files to avoid thousands of micro-files.

Claude focused more on edge identity and immutability than physical storage.

Synthesis:

> Logical EDGE identity is required. Physical storage may be one file per edge or append-only ledger files.

The storage form should not define the ontology.

Acceptable bootstrap options:

```text
docs/edges/EDGE-0001.yaml
```

or:

```text
docs/edges/P-0001.edges.yaml
```

or later:

```text
graph-managed edge store
```

Requirement:

> Each logical edge must remain independently addressable and immutable.

**Decision:** Individual edge files are acceptable for initial simplicity; ledger storage is deferred as an optimization.

---

## 10. Structural vs Epistemic Edges

Gemini proposed distinguishing structural edges from epistemic edges.

This distinction is plausible but not necessary for the immediate implementation proposal.

Possible future metadata:

```yaml
edge_class: structural
```

or:

```yaml
edge_class: epistemic
```

Examples:

- structural: `supersedes`, `implements`, `reviews`
- epistemic: `supports`, `disputes`, `warrants`, `contradicts`

**Decision:** Deferred as speculative but worth preserving.

---

## 11. Prefix Discipline

Claude recommended keeping birth-stamp IDs but declaring prefix meanings closed and non-redefinable.

Synthesis accepts:

> Prefixes are origin birth-stamps. They are never redefined.

If a new family is needed, append a new prefix. Do not reinterpret an existing prefix.

**Decision:** Accepted with moderate-high confidence.

---

## 12. Catalog Generation

Claude recommended generating the catalog from the start, even with a trivial script, to avoid drift.

Synthesis accepts as implementation guidance:

> The catalog should be generated from front matter and edge records as soon as practical.

The first implementation may still begin with manual files, but the validator/generator should be introduced early.

**Decision:** Accepted with moderate confidence.

---

## 13. Required Decision Before Implementation

Before writing many edge records, AI-Lab should freeze a focused decision:

```text
ADR-0003: Edge Identity, Immutability, and Addressing
```

This ADR should define:

1. Edge records are immutable.
2. Edge correction happens by new edge creation.
3. Edges can target edges.
4. References include explicit scope: `version`, `lineage`, or `exact`.
5. Prefixes are immutable birth-stamps.
6. Catalogs are derived indexes.
7. Physical edge storage is an implementation detail.

---

## 14. Should P-0001 Become v0.4?

Two options exist.

### Option A — Write ADR-0003 First

Keep P-0001 v0.3 as the candidate implementation proposal, and create ADR-0003 to freeze the edge model.

Advantages:

- avoids endlessly expanding P-0001;
- keeps the edge decision focused;
- lets implementation begin after ADR-0003.

### Option B — Produce P-0001 v0.4

Revise P-0001 again to include the edge model directly.

Advantages:

- one proposal contains the complete bootstrap design;
- easier for reviewers to read in one place.

### Synthesis Recommendation

Prefer Option A.

P-0001 has reached sufficient maturity as a proposal. The remaining problem is now narrow enough to become an architecture decision.

**Decision:** Prefer ADR-0003, not P-0001 v0.4.

## 16. Closing Note

The third review cycle produced a meaningful architectural discovery:

> Nodes and edges are both artifacts. If nodes can be frozen, edges must be immutable too. The graph evolves not by rewriting history, but by adding warranted relations to prior relations.

This completes the conceptual bridge from Markdown files to a future graph-native AI-Lab.
